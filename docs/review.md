# Architecture Review — agent_rnaseq

Reviewed files: `CLAUDE.md`, `docs/architecture.md`, `docs/specs/*.md`, `tasks/*.md`

---

## Critical Issues

### C1 — Missing `SplicingResult` database model
`data_models.md` has `DEGResult`, `GSEAResult`, and `VariantCall` tables but no `SplicingResult` table. The `splicing_table` artifact type exists, `RMATSOutput` produces event counts per type, and `SplicingAgent` is a first-class stage — yet there is no persistent storage for individual splicing events (SE, A5SS, A3SS, MXE, RI rows). This means splicing results cannot be queried via API, and the Streamlit viz layer has no structured data to consume.

**Fix**: Add `SplicingResult` to `data_models.md` with fields: `id`, `stage_id`, `run_id`, `contrast`, `event_type` (SE/A5SS/A3SS/MXE/RI), `gene_id`, `gene_name`, `inclusion_level_diff`, `fdr`, `pvalue`. Add `GET /runs/{run_id}/splicing` to `api_contracts.md`.

---

### C2 — Missing `GET /runs/{run_id}/splicing` and `GET /runs/{run_id}/variants` API endpoints
`api_contracts.md` has `/qc`, `/de`, `/gsea` result endpoints but omits endpoints for the two other result types with DB tables: splicing (`SplicingResult`) and variants (`VariantCall`). Users cannot query these results without direct DB access.

**Fix**: Add both endpoints to `api_contracts.md` with schemas consistent with the existing result endpoints.

---

### C3 — `BackgroundTasks` is wrong executor for long-running pipeline runs
TASK-11 specifies `POST /runs` queues the pipeline via FastAPI `BackgroundTasks`. RNA-seq runs last minutes to hours; `BackgroundTasks` executes in the same process as the web server and has no persistence across restarts, no retry logic, and no visibility into queue depth. If the API process restarts, all pending runs are silently lost.

**Fix**: Replace `BackgroundTasks` with a durable task queue. Options in scope: ARQ (async, Redis-backed), Celery, or AWS SQS triggering a worker. Add a `Worker` service to `docker-compose.yml`. Update TASK-11 acceptance criteria. If keeping `BackgroundTasks` for initial dev, document explicitly that it is dev-only and add a prod-readiness task.

---

### C4 — LangGraph + OpenAI Agents SDK boundary is undefined
The architecture says "Orchestrator Agent (OpenAI Agents SDK + LangGraph)" but never specifies how the two frameworks interact. LangGraph already owns routing, state, and checkpointing. The OpenAI Agents SDK also provides agent loops and tool-calling. Using both without a clear boundary will result in duplicated logic and undefined behavior (e.g., which framework owns the retry loop when a tool fails?).

**Fix**: Add a section to `docs/architecture.md` defining the exact boundary:
- LangGraph owns: DAG routing, state schema, checkpointing.
- OpenAI Agents SDK owns: LLM calls for intent parsing and stage summaries.
- LangGraph nodes call OpenAI SDK client directly (not as an "agent loop") for single-turn completions.

Update TASK-09 to reflect this split explicitly.

---

### C5 — No mock execution mode specified
TASK-15 says "Zero real subprocess calls to bioinformatics tools (all mocked)" but there is no central spec for how tools are mocked. Each test file will implement ad-hoc mocking, leading to inconsistent stubs and missed coverage. More importantly, there is no `dry_run` flag in `RunConfig` or a `MockExecutor` for interactive testing without bioinformatics tool installations.

**Fix**: Add `dry_run: bool = False` to `RunConfig` (enforced via Pydantic, never set by LLM — must be set in API request). Define a `MockToolRegistry` in `docs/specs/tool_contracts.md` that maps each tool name to a canned `ToolOutput` fixture. Update TASK-01 to create `tests/fixtures/mock_tool_outputs/` with per-tool JSON fixture files. Update TASK-15 to reference this registry.

---

### C6 — `compile_report` input is untyped `dict` — LLM summaries can enter report without validation
`tool_contracts.md` defines `ReportInput.qc_summary: dict | None` and `de_summary: dict | None`. These dicts are generated by agents after LLM summary calls, meaning LLM-generated text can flow directly into the report tool without schema enforcement. This violates the safety policy (Rule 2: no raw LLM text as a data source).

**Fix**: Replace `dict` types with typed Pydantic models: `QCSummaryForReport`, `DESummaryForReport`, `GSEASummaryForReport`. Each model accepts only fields derived from validated `StageOutput` objects. LLM-generated narrative text (if any) goes into a separate `narrative: str | None` field that is clearly marked as non-authoritative in the report template.

---

### C7 — `Aligner` enum name is misleading — conflates alignment and quantification tools
`data_models.md` defines `Aligner: star | salmon | rsem`. Salmon and RSEM are quantification tools, not aligners. STAR is the aligner; Salmon/RSEM perform quantification. The enum name causes confusion about what `AnalysisRun.aligner` means and whether alignment is even required when `aligner=salmon`.

**Fix**: Rename `Aligner` to `QuantificationMethod: star_htseq | salmon | rsem`. Add a separate `aligner: Literal["star"]` field (currently the only supported aligner). Update all references in `api_contracts.md`, TASK-02, TASK-04, TASK-05, and TASK-09.

---

### C8 — Stage dependency enforcement is not specified
`POST /runs` accepts a `stages` list. Nothing prevents a user or LLM from submitting `stages=["de"]` without `quantification`, which would cause `DEAgent` to fail looking for a counts matrix. There is no spec for stage dependency validation.

**Fix**: Add a `STAGE_DEPENDENCIES` map to `docs/specs/tool_contracts.md`:
```
alignment   → requires: qc (optional but warned)
quantification → requires: alignment
differential_expression → requires: quantification
gsea        → requires: differential_expression
splicing    → requires: alignment
variant_calling → requires: alignment
```
Add a validation step in `OrchestratorAgent` that rejects `RunConfig` when stages violate dependencies. Add acceptance criteria for this to TASK-09.

---

### C9 — No `APIKey` model — authentication spec is incomplete
`api_contracts.md` states auth via Bearer token but `data_models.md` has no `APIKey` entity. TASK-11 says keys are "stored in DB or env var list" without committing to either. Without a DB model, there is no key rotation, no per-key audit trail, and no admin API to issue or revoke keys.

**Fix**: Add `APIKey` table to `data_models.md`: `id`, `key_hash` (SHA-256, never plaintext), `name`, `created_at`, `expires_at`, `revoked_at`, `created_by`. Add `POST /api-keys` and `DELETE /api-keys/{id}` endpoints to `api_contracts.md`. Record `api_key_id` on `AnalysisRun.created_by` (change type from `VARCHAR(64)` to FK).

## Recommended Changes

### R1 — Add `PipelineStage.exit_code` column
`safety_policy.md` Rule 9 says exit code is stored, but `data_models.md`'s `PipelineStage` only has `error_message`. Add `exit_code: INTEGER` to `PipelineStage`. This makes failure diagnosis deterministic and is already referenced in `ToolExecutionError`.

---

### R2 — Specify genome ID resolution for natural-language input
TASK-09 says the Orchestrator rejects an unknown `genome_id`, but does not specify how the user's natural-language "use GRCh38" gets converted to a DB UUID. The LLM must look up genomes from a list, but no spec covers this step. Add a "Genome Resolution" section to `docs/architecture.md`: the Orchestrator must call `GET /genomes` (or query DB directly) and present options to the user if ambiguous, rather than guessing a UUID.

---

### R3 — Add `GET /projects/{project_id}` and `GET /samples/{sample_id}` endpoints
`api_contracts.md` only defines list endpoints for projects and samples. Clients need to fetch individual resources. Add single-resource GET endpoints consistent with REST conventions.

---

### R4 — Specify WebSocket log collection architecture for Nextflow/AWS Batch
`architecture.md` describes a WebSocket endpoint for real-time log streaming, but logs for Nextflow jobs come from subprocess stdout and AWS Batch jobs come from CloudWatch. There is no spec for how these are collected and forwarded to WebSocket clients. Add a "Log Aggregation" section to `architecture.md` specifying a polling or push strategy (e.g., Nextflow stdout → Redis pub/sub → WebSocket, CloudWatch → polling → WebSocket).

---

### R5 — Add `scRNAClusterResult` model for Scanpy output
`ScanpyOutput.cluster_summary` contains `n_clusters` and `cells_per_cluster` but this is stored only in `PipelineStage.output_summary` JSONB. There is no structured table for single-cell results. Add a lightweight `scRNAClusterResult` table to `data_models.md` so Streamlit can query cluster statistics without parsing JSONB.

---

### R6 — Pagination missing from `GET /genomes` and `GET /projects`
`GET /runs` has `limit`/`offset` but `GET /genomes` and `GET /projects` return unbounded arrays. Add `limit`/`offset` (or cursor-based) pagination to both endpoints in `api_contracts.md`.

---

### R7 — Add splicing integration test to TASK-15
`test_bulk_rnaseq_pipeline.py` in TASK-15 covers QC → alignment → quant → DE → GSEA but omits splicing. Add a `test_splicing_pipeline.py` covering alignment → rMATS → `SplicingResult` DB writes.

---

### R8 — `AnalysisRun.created_by` should be NOT NULL
`data_models.md` marks `created_by` as nullable with no comment. Every run must be attributable to an API key or user. Change to NOT NULL once the `APIKey` model (C9) is in place.

---

### R9 — Add `tool_version` recording test to TASK-15
Safety Rule 8 requires tool versions to be recorded in `PipelineStage.tool_version`. No acceptance criterion in any task verifies this. Add: "Each completed `PipelineStage` has a non-null `tool_version`" to TASK-15 integration test criteria.

---

### R10 — Document `alignment_mode=both` quantification behavior
`AnalysisRun.alignment_mode` has value `both` (genome + transcriptome), but no spec describes which count matrix feeds DE analysis when both are produced (STAR GeneCounts vs. Salmon/RSEM). Add a decision table to `docs/specs/data_models.md` or `docs/architecture.md`.

---

## Optional Improvements

### O1 — Consider collapsing `VizAgent` and `ReportAgent` into tool-only steps
`VizAgent` calls `prepare_streamlit_data` and `generate_ucsc_tracks`; `ReportAgent` calls `compile_report`. These are data transformation/IO steps with no routing logic, no LLM call, and no branching. Implementing them as full agent classes (with `BaseStageAgent` overhead) may be over-engineering. Consider making them terminal tool calls invoked directly from the LangGraph END node rather than separate agent classes.

---

### O2 — Add observability spec (metrics + tracing)
There is no mention of metrics (Prometheus/Datadog), distributed tracing (OpenTelemetry), or alerting. For a production pipeline service, at minimum define: run duration histogram, stage failure rate counter, active run gauge, and API latency p95. Add a `docs/specs/observability.md` stub and a `TASK-17` placeholder.

---

### O3 — Add production secrets management note
`docs/specs/safety_policy.md` Rule 6 covers credentials, but there is no spec for secrets rotation, vault integration (HashiCorp Vault, AWS Secrets Manager), or K8s Secrets. Add a note to `docs/deployment.md` (TASK-16) covering this before considering the system production-ready.

---

### O4 — `DEGResult` table will be very large — add index and archiving guidance
A typical RNA-seq run produces ~20k `DEGResult` rows per contrast. At 10 runs × 3 contrasts, that's 600k rows; at scale this grows quickly. Add composite indexes on `(run_id, contrast, padj)` to `data_models.md`. Note that old runs' result rows should be archivable. Add this to TASK-02 acceptance criteria.

---

### O5 — Pin LLM model in config — do not rely on SDK defaults
No spec names the OpenAI model used by the Orchestrator or specialist agents. SDK defaults change over time and could silently alter behavior. Add `AGENT_LLM_MODEL: str = "gpt-4o"` to `src/config.py` spec in TASK-09, validated against an allowlist.

---

### O6 — Add property-based tests for tool output parsers
Tool output parsers (STAR log parser, FastQC summary parser, DESeq2 CSV parser) are high-risk parsing code. Consider adding hypothesis-based fuzz tests to TASK-03–TASK-07 that verify parsers handle malformed output gracefully rather than silently producing wrong values.

---

### O7 — `SplicingAgent` is missing from the agent routing diagram in `architecture.md`
Section 2.2 of `architecture.md` lists the Splicing Agent in the agent table but the routing diagram in Section 3 has no `[splicing]` edge. Add `├──[splicing]──→ Splicing Agent node` to the diagram.
